Section 03  ·  Technology & Dependency Audit

What Breaks the Requirements

The good news: this file contains zero proprietary platforms. No Firebase, no Databutton, no Riff, no vendor-locked backend. All logic is native HTML/CSS/JS. Here are the specific dependency flags and their resolutions:


  • ⚠ External Dependency — Google Fonts CDN Two Google Fonts imports loaded via <link> from fonts.googleapis.com. While not vendor-locked in the problematic sense, a production app should self-host fonts for GDPR compliance, performance, and offline capability. Self-host via @font-face declarations with locally bundled .woff2 files. Use `fontsource` npm packages in the React build.
  • ⚠ Browser API — Web Speech API (SpeechSynthesis) The narrator engine uses window.speechSynthesis — a browser-native API with inconsistent cross-browser support (limited Firefox, unreliable on mobile Safari, Chrome-only for boundary events). No server-side TTS whatsoever. Replace with a real TTS API: ElevenLabs, AWS Polly, or Google Cloud TTS. Pre-generate audio files per chapter, store in S3/object storage, stream via FastAPI endpoints. Far more consistent and enables offline/cached playback.
  • ⚠ No Persistence — Content Hardcoded in HTML All story content — chapters, titles, disclaimers — is hardcoded directly in the HTML. There is no CMS, no database, no API. Editing a single paragraph requires opening the HTML file. Move all story content into PostgreSQL (books, chapters, paragraphs tables). Serve via FastAPI. Render dynamically in React. Enables editing, versioning, multi-story support.
  • ⚠ No Auth / No Reader State No user identity, no reading progress saved, no bookmarks, no chapter completion tracking. The narrator state is ephemeral — refreshing loses your place. Add JWT auth (FastAPI), a reading_progress table in PostgreSQL, and a React context for reader state. Persist chapter position and narrator playback index to the DB.
  • ⚠ Embedded Binary Asset — Base64 Silhouette PNG The silhouette figure is a 15,000+ character base64 PNG embedded directly in the CSS. This inflates page weight and is unmaintainable. Extract to a proper image file. Serve from a CDN or static file server. Reference via standard <img> or CSS background-image url().
  • ⚠ No SEO / Meta / OG Tags Zero Open Graph tags, no Twitter card, no canonical URL, no <meta description>. The press-release framing makes this a shareable asset — but it can't be shared effectively without social preview metadata. Add full OG/Twitter meta. Generate dynamic og:image previews server-side per book. Use FastAPI + React Helmet / next/head equivalent.

Section 04  ·  Missing Functional Requirements

The Gap List

This file is a beautiful, polished front-end demo. What it is not yet is a product. The gap between these two things is precisely what this section defines.

App Audit Team
  • Authentication & Authorization No login, no reader accounts, no author/admin separation. Need: JWT auth, role-based access (reader, author, admin), email verification flow, password reset.
  • Database Models Required Users, Books, Chapters, Paragraphs, ReadingProgress, NarratorSettings, AudioFiles, Comments (optional), PublishStatus. Minimum 6 PostgreSQL tables to support a real product.
  • Backend API Endpoints GET /books/:id, GET /books/:id/chapters, GET /chapters/:id, GET /chapters/:id/audio, POST /progress, GET /progress/:user/:book, PUT /users/:id/preferences. Minimum 8 REST endpoints.
  • Real TTS / Audio Pipeline Web Speech API is a demo. A real product needs server-side TTS generation (ElevenLabs / AWS Polly), audio file storage (S3), signed URL delivery, and chapter-level audio caching. This is the single highest-complexity gap.
  • Content Management Authors need a way to write, edit, reorder, and publish chapters. Minimum: a basic admin CRUD interface for Books and Chapters. Stretch: a rich-text editor with dark theme.
  • Frontend State Management Narrator state (current chapter, position, speed), reading progress, user auth state all need React context or a lightweight store (Zustand). Currently all state is lost on page refresh.
  • Error Handling & Loading States Zero error states, zero loading skeletons. A production app needs: chapter load failures, audio fetch timeouts, auth errors, and graceful fallbacks throughout.
  • Monetization Infrastructure No payment layer exists. To charge for content: Stripe integration, subscription tiers (free preview vs. full access), or one-time purchase per book. Requires a subscriptions/purchases table and webhook handler.
  • Security No CORS policy, no rate limiting, no input sanitization, no auth guards on content endpoints. FastAPI middleware layer needed before any public deployment.

Section 06  ·  Implementation Plan

Step-by-Step Roadmap

1
Extract & Convert UI to React Components

Break the single HTML file into React components: Cover, TableOfContents, ChapterCard, PullQuote, NarratorPlayer, Footer. Preserve all visual styling (CSS modules or Tailwind). Self-host Google Fonts via fontsource. Extract base64 image to a proper asset file.

2
Define Data Contracts (Pydantic Schemas)

Model Book, Chapter, User, AudioFile, ReadingProgress as Pydantic schemas in FastAPI. Define request/response shapes for every endpoint before writing any route logic.

3
Build PostgreSQL Models & Migrations

Use SQLAlchemy ORM with Alembic for migrations. Seed the existing U.G.L.Y. story content as initial data. Establish foreign key relationships between books → chapters → audio_files.

4
Implement FastAPI Routes & Business Logic

Build CRUD routes for books, chapters, and reading progress. Add JWT authentication middleware. Implement rate limiting. Wire up CORS for the React frontend origin.

5
Build Real TTS Audio Pipeline

Integrate ElevenLabs or AWS Polly. On chapter publish, trigger async audio generation via a background task (FastAPI BackgroundTasks or Celery). Store .mp3 files in S3. Return signed URLs to the frontend. Replace the Web Speech API narrator entirely.

6
Connect Frontend to Backend (Real API Requests)

Replace hardcoded HTML content with API fetches. Implement React context for auth state and reading progress. Wire the narrator player to real audio URLs from the backend. Persist chapter position on advance.

7
Add Auth, Validation & Monetization

Build login/register flows. Protect chapter content behind auth guards. Integrate Stripe for subscriptions. Add a free-preview model (first 2 chapters free, rest paywalled). Handle Stripe webhooks in FastAPI.

8
QA & Repo Readiness

Write Pytest unit tests for all FastAPI routes. Test React components with Testing Library. Add README with setup instructions, env variable documentation, and seed data script. Add Dockerfile + docker-compose for local development. Verify the full read→narrate→progress flow end-to-end.
