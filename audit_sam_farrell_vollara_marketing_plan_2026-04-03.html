Section 03 · Technology & Dependency Audit

What Breaks the Requirements

Zero proprietary platforms. No Firebase, no Databutton, no Riff, no vendor-locked services. Pure HTML/CSS/JS. This is a clean base to build from. The violations below are structural and architectural — not platform lock-in issues.

  • ⚠ External Dependency — Google Fonts CDN Bebas Neue, DM Sans, and DM Mono loaded from fonts.googleapis.com. Not vendor lock-in, but a GDPR / privacy compliance concern for EU users, and a performance dependency on an external CDN. Self-host all three fonts via @font-face with locally bundled .woff2 files. Use fontsource npm packages in the React build pipeline.
  • ⚠ Zero Data Persistence — Entire Plan is Hardcoded All content — phases, scripts, specs, revenue projections, KPIs, timelines — is hardcoded directly in HTML markup. Sam cannot update a sales script or change a KPI target without editing raw HTML. There is no CMS, no admin panel, no editable data layer whatsoever. Move all plan content into PostgreSQL (plan_sections, scripts, kpis, phases, channels tables). Serve via FastAPI. Render dynamically in React. Enables editing, versioning, and multi-associate use.
  • ⚠ No Authentication — Document is Fully Public The document is labeled "Confidential · Internal Use Only" but has zero access control. Anyone with the URL can read it. No login, no password protection, no token gating. Add JWT auth in FastAPI. Gate the full plan behind a login. Store user sessions in PostgreSQL. Add role: associate vs. admin for multi-user scenarios.
  • ⚠ No Progress Tracking or Task Completion The 90-day roadmap and phase action plan are purely read-only. Sam cannot check off tasks, mark phases complete, or track which sales scripts he's deployed. All execution tracking happens in his head or in a separate tool. Add a task_completions table in PostgreSQL. Build a React checkbox/toggle system per step-card. Persist completion state to the backend. Show a progress bar per phase.
  • ⚠ No CRM / Lead Management Layer The plan recommends building an email list to 50+, running demos, tracking prospects, and managing referrals — but provides zero tooling for any of it. There is no lead table, no contact log, no demo scheduler, no pipeline tracker. Build a lightweight CRM module: leads table (name, contact, status, source), demo_log table, referral_tracking table. FastAPI CRUD endpoints. React kanban or list view. This alone transforms the tool from a document into a product.
  • ⚠ Revenue Table is Static — No Live Calculation The revenue projection table is hardcoded HTML. Sam cannot adjust commission rate, unit price, or scenario count and see updated projections. It's a snapshot, not a calculator. Replace with a React-based interactive revenue calculator. Input fields for units sold, retail price, commission %, and referral bonuses. Live calculation via React state. Store scenarios in PostgreSQL for comparison.
  • ⚠ No SEO / OG / Social Sharing Meta Zero Open Graph tags, no twitter:card, no meta description, no canonical URL. If Sam ever wants to share a specific section (the sales scripts page, the product spotlight) via social, there is no preview card. Add full OG/Twitter meta via React Helmet. Generate dynamic og:image per section. Use FastAPI to serve dynamic meta for section-specific URLs.
  • ⚠ No Mobile Optimization for Script Use Case Sam's primary in-field use case is likely pulling up a sales script on his phone mid-conversation. The responsive CSS is functional but the script boxes are not optimized for fast mobile retrieval — no search, no quick-access shortcuts, no copy-to-clipboard functionality on scripts. Add a React "Scripts Mode" view: searchable, filterable script library with copy-to-clipboard, favoriting, and a mobile-first card layout. Store favorited scripts per user in PostgreSQL.

Section 04 · Missing Functional Requirements

The Full Gap List

This file is a beautifully designed static document. What it needs to become a real sales operations platform is everything that sits behind the presentation layer.

Audit Assessment
  • Authentication & Authorization No login system. Need: JWT auth, user registration, password reset, role-based access (associate vs. admin/coach), session persistence in PostgreSQL.
  • Database Schema Required Minimum tables: users, plan_sections, phases, step_cards, scripts, objections, channels, kpis, revenue_scenarios, leads, demo_log, task_completions, referrals. Approximately 12 PostgreSQL tables for a full operational platform.
  • Backend API Endpoints Required GET /plan, GET /scripts, GET /phases, GET /kpis, POST /leads, PUT /leads/:id, POST /tasks/complete, GET /revenue/calculate, GET /demos, POST /demos. Minimum 10+ FastAPI endpoints.
  • CRM / Lead Management Module The single highest-value addition. A lightweight pipeline for tracking prospects from cold outreach → demo → follow-up → closed/lost. Kanban-style board or list view. This is the gap between a document and a business tool.
  • Task Completion & Progress Tracking Every step-card in the 6-phase plan needs a checkable state. Every phase needs a completion percentage. A dashboard showing Sam's overall 90-day progress. Currently none of this exists.
  • Interactive Revenue Calculator Replace the static HTML table with a live React calculator: adjustable unit count, price, commission rate, referral multiplier. Show monthly and annual projections. Store named scenarios for comparison.
  • Script Library with Search & Copy A filterable, searchable scripts module with copy-to-clipboard, favorite/bookmark, and usage tagging (e.g., "used 3 times"). Critical for mobile field use.
  • Content Calendar / Social Scheduling The plan recommends TikTok 1x/day, Facebook posts, Instagram Reels, email newsletters — but provides no calendar, no scheduler, no content planning grid. A simple content_calendar table + React calendar view would bridge this.
  • KPI Dashboard with Real Inputs Current KPIs are decorative static numbers. A real version lets Sam input actuals (demos run, units sold, email list size, referrals generated) and compare against targets in real time.
  • Error Handling, Loading States, Mobile UX Zero error states, zero loading skeletons, no offline capability. A field sales tool needs graceful handling of poor mobile connections and fast script retrieval.

Section 06 · Implementation Plan

Step-by-Step Roadmap

1
Componentize the React UI — Port All Sections

Break the single HTML file into React components: HeroSection, ProductSpotlight, PhaseList, PhaseCard, StepCard, ScriptBox, ObjectionCard, RevenueTable, KPIStrip, ChannelGrid, TimelineBar. Self-host all three Google Fonts. Preserve every visual style exactly — the design is strong and should be kept.

2
Stand Up FastAPI + PostgreSQL Skeleton

Initialize FastAPI project with SQLAlchemy + Alembic. Define all 12 tables. Write initial migration. Seed all existing plan content (phases, step cards, scripts, objections, channels, KPIs) from the HTML file into the database. Write GET endpoints to serve all plan content dynamically.

3
Add JWT Authentication

Build login/register endpoints in FastAPI. Add JWT middleware. Create the users table with role column. Protect all write endpoints behind auth guards. Build a minimal React login screen. Store session in localStorage or httpOnly cookie.

4
Build Task Completion & Progress Tracking

Add checkboxes to every StepCard component. Wire to POST /tasks/complete and DELETE /tasks/complete endpoints. Compute phase completion % server-side. Render a progress bar per phase and overall 90-day completion dashboard. This is the highest immediate-value feature for Sam.

5
Build the CRM / Lead Pipeline Module

Create a leads management page with Kanban columns: Cold → Demo Scheduled → Demo Done → Follow-Up → Closed Won / Lost. Each card shows lead name, contact method, source, and notes. POST /leads, PUT /leads/:id/status, POST /demos. This turns the document into a live business tool.

6
Build Interactive Revenue Calculator + Live KPI Dashboard

Replace the static revenue table with a React calculator (sliders + inputs for units, price, commission %, referrals). Replace static KPI strip with input fields for actuals. Save actuals to kpi_actuals table. Show target vs. actual comparison with progress indicators.

7
Build Mobile-First Script Library

Dedicated /scripts page with search bar, category filter (DM · In-Person · Objection · Follow-Up), copy-to-clipboard per script, and favorite/bookmark system. Optimized for one-handed mobile use mid-sales-conversation. Store favorites in PostgreSQL per user.

8
QA, Seed, README + GitHub Readiness

Write Pytest tests for all FastAPI routes. Test React components with Testing Library. Add comprehensive README with setup, env variables, migration commands, and seed data instructions. Add Dockerfile + docker-compose.yml. Verify full auth → plan → tasks → leads → CRM flow end-to-end.
