Technology Audit · Section 03

Violations & Flags

Everything that breaks the React + FastAPI + PostgreSQL mandate — or signals critical functional debt.

⚠ Clean Base
Zero Prohibited Platform Dependencies
Unlike other audited files in this series, The Prostate Truth HTML is completely clean of Firebase, Databutton, Riff, or any vendor-locked platform code. All CSS is inline, all fonts loaded from Google Fonts, zero JS frameworks. Fully portable.
⚠ Technical Debt
Dead CTA Buttons — No Functional Actions
Both CTAs in the final section point to href="#". "Launch the Campaign" and "Download the Full Plan" produce zero outcomes. In a production app these need either a campaign waitlist form (FastAPI POST → PostgreSQL subscribers table) or a PDF generation endpoint (WeasyPrint/ReportLab via FastAPI).
Replacement: FastAPI POST /api/subscribers for email capture + React modal form; FastAPI GET /api/plan/download for server-rendered PDF export.
⚠ Static Content
All Data is Hardcoded HTML — No CMS or Database
Every stat, avatar story, expert bio, pillar description, and timeline phase is hardcoded in HTML. Any update requires editing the source file directly. As the platform scales content across 8 channels and 4 campaign phases, this becomes unmanageable in hours, not years.
Replacement: PostgreSQL content tables + FastAPI CRUD endpoints + React admin dashboard. All content becomes database records editable without redeployment.
⚠ No Personalization
No User State, Auth, or Screening Tracker
The file implies — and the plan explicitly calls for — a mobile app with PSA tracking reminders, appointment scheduling, and personalized content feeds. None of that infrastructure exists. There's not even a login button.
Replacement: JWT-based auth (FastAPI + OAuth2 + bcrypt), PostgreSQL users table, screening_records table, appointments table, and a React dashboard with personal health timeline.
⚠ No Community Features
Community Forum Mentioned, Not Built
The website/blog channel description explicitly mentions a "community forum." No forum exists. This is a significant engagement feature gap — community is central to the Sam & Lamar brand.
Replacement: PostgreSQL forum_posts + forum_threads + forum_members tables; FastAPI discussion endpoints; React threaded comment components.
Gap Analysis · Section 04

What It Needs to Become Real

The full gap list — authentication, database schema, API layer, and features required to transform this gorgeous static plan into a living, functional advocacy platform.

  • 01
    Authentication & User Identity — No login, registration, or user identity exists. Needed for PSA tracking, content personalization, community forum, ambassador enrollment, and event registration. Requires JWT + OAuth2 + bcrypt in FastAPI and a users table with role columns (visitor, member, ambassador, admin).
  • 02
    Content Management System — All articles, pillar content, expert bios, and stats are frozen in HTML. Needs a full CRUD content API and React admin panel so non-technical staff can publish blog posts, update statistics, add expert advisors, and manage campaign phases without touching code.
  • 03
    PSA Screening Tracker — The plan explicitly calls for a mobile app with PSA tracking reminders. Needs a screening_records table (user_id, psa_value, test_date, provider, notes), a FastAPI series of endpoints, and a React timeline chart showing PSA velocity over time.
  • 04
    Email Capture & Newsletter System — The plan targets email list building as a primary growth lever. CTA buttons are dead. Needs a subscribers table, a FastAPI POST /subscribe endpoint, email validation, and integration with an email sending service (SendGrid/Mailgun via SMTP relay).
  • 05
    Community Forum — Explicitly referenced in the website/blog section. A fully featured discussion system with thread creation, replies, moderation, and member profiles. PostgreSQL: forum_categories, forum_threads, forum_posts, forum_members.
  • 06
    Ambassador / Partner Enrollment — Phase 2 calls for a "Barbershop Partnership Program" and "Church Outreach Kit." No form or workflow exists. Needs an application form → admin review → approval pipeline with status tracking and automated outreach kit delivery.
  • 07
    Doctor Finder / Locator — The website description mentions a "local doctor finder." Needs either a curated provider database (PostgreSQL: providers table with specialty, location, geocoords) with a React map interface, or integration with NPI lookup APIs.
  • 08
    Event Registration System — Phase 4 launches an Annual Health Summit (virtual + in-person). Needs a events table, registration workflow, capacity management, confirmation emails, and a virtual session integration (Zoom API or similar).
  • 09
    Podcast & Video CMS — The podcast is the anchor content format. Needs an episodes table, audio file storage (S3 or Cloudflare R2), embedded player, show notes, and an RSS feed generator for podcast directories (Apple, Spotify).
  • 10
    Admin Dashboard & Analytics — No admin tooling exists. Needs subscriber counts, content views, PSA tracker usage, forum activity, ambassador applications, and campaign milestone tracking in a React admin panel backed by FastAPI aggregate endpoints.

Implementation Roadmap · Section 06

Build Sequence

A practical, phased engineering sequence to transform this advocacy plan into a production-ready platform. Ship value early, build authority incrementally.

Step 01 — Week 1–2
Project Scaffolding & Database Schema
Create the monorepo structure (frontend/, backend/, docs/). Initialize FastAPI with SQLAlchemy async + Alembic. Define all PostgreSQL models. Configure environment variables. Set up CI/CD pipeline (GitHub Actions).
Step 02 — Week 2–3
Authentication Layer
JWT auth with FastAPI OAuth2PasswordBearer. User registration, login, email verification, password reset. React AuthContext with protected routes. Role-based access control (visitor / member / ambassador / admin).
Step 03 — Week 3–5
Convert HTML to React Component Library
Translate every HTML section into React components using the existing design system (color vars, typography, grid patterns). Build reusable: AvatarCard, StatBox, ContentPillarCard, PillarGrid, TimelineItem, ChannelGrid. Replace all hardcoded content with props.
Step 04 — Week 5–7
Content API & CMS Admin Panel
FastAPI CRUD routes for blog posts, pillar content, expert records, campaign phases, and stats. React admin dashboard with rich text editor (TipTap). Connect frontend to API via React Query. Replace all hardcoded HTML content with API-driven data.
Step 05 — Week 7–9
PSA Tracker & Screening Features
Build the PSA screening log (the platform's highest-value clinical feature). React form for PSA entry, Recharts timeline visualization showing PSA velocity over time, reminder scheduling via Celery Beat, and export to PDF. Authenticated route, user-specific data only.
Step 06 — Week 9–11
Email Capture, Newsletter & Ambassador Enrollment
Live CTA buttons: email subscribe form → FastAPI → PostgreSQL subscribers table → SendGrid confirmation email. Ambassador application form → admin review queue → approval workflow → automated outreach kit delivery (PDF email attachment).
Step 07 — Week 11–13
Podcast Player & Media Pipeline
S3/R2 audio upload in admin. FastAPI episode endpoints. React custom audio player (waveform visualization optional). Auto-generated RSS feed for podcast directory submission. YouTube embed integration for video content.
Step 08 — Week 13–15
Community Forum & Doctor Finder
PostgreSQL-backed discussion forum with thread/post/reply model. Moderation tooling in admin. Doctor finder with NPI API integration or curated provider database, React map view (Google Maps or Leaflet), and filter by specialty and accepting status.
Step 09 — Week 15–16
QA, Security Hardening & Repo Readiness
Input validation on all endpoints (Pydantic). Rate limiting (slowapi). HIPAA-awareness review for any health data storage. Security headers. README with setup instructions. Docker Compose for local dev. Deployment scripts for Render/Railway/Fly.io. End-to-end test coverage for critical paths.

This is the strongest advocacy marketing asset audited in this series — editorially sophisticated, culturally intelligent, and structurally complete as a brand plan. It is also the most ambitious in terms of functional scope. The gap between "stunning static HTML" and "functioning national health platform" is large but entirely well-defined.
