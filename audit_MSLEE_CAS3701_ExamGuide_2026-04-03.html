Overview · Section 01

What This File Is

MSLEE_CAS3701_ExamGuide.html is a sophisticated, single-file static HTML academic study guide produced for a university-level accounting and business module — CAS3701, a South African postgraduate programme examination due 7 April 2026. The file is not a marketing page, a SaaS product, or an onboarding flow. It is an educational content delivery artifact — a beautifully designed, exam-specific reference tool authored by someone identified as "Ms. Lee."

The file covers two major examination questions: Question 1 addresses consolidated financial reporting under IFRS 10/3 and IFRS 15 (Mabaso Health Holdings Ltd scenario), and Question 2 covers business structure advisory, strategic analysis, and start-up cost planning for a fictional e-commerce wellness brand (WellCore / BrightPath Consulting). The guide is exceptional in quality — it features theoretical model answers, question-by-question breakdowns, examiner coaching notes ("Your Writing Coach Says"), SWOT grids, comparison tables, and IFRS-anchored critique structures.

"This is not a demo page. It is a precision-engineered academic coaching tool — and one of the more intellectually rich HTML files audited in this series. The product hiding inside it is a full-featured EdTech study platform."

Core Product

Exam preparation platform — structured study guides with model answers, coaching commentary, and structured frameworks for professional-level accounting examinations.

Implied User Roles

Student (primary consumer of content), Educator/Author (Ms. Lee — content creator), and potentially an Admin role for managing guide libraries across modules and cohorts.

Implied User Journey

Student receives/accesses guide → navigates by question section → reads model answers → applies coaching tips → prepares submission. No auth, no persistence, no progress tracking.

Academic Domain

South African higher education accounting — IFRS, Companies Act 71 of 2008, Income Tax Act, POPIA. Content is jurisdiction-specific and professionally authoritative.

Technology Audit · Section 03

Violations & Dependency Flags

No prohibited platform code was detected. The file is remarkably clean — pure HTML, CSS, and Google Fonts. Zero Firebase, zero Databutton, zero Riff, zero vendor-locked backend. The violations that follow are architectural gaps rather than platform contamination.

✓ No Firebase. No Databutton. No Riff. No embedded widget. No proprietary platform dependencies detected. Clean slate.
Violation 01 Zero Backend — Fully Static HTML

The entire file is a static HTML document with no server-side logic, no API calls, no authentication layer, and no data persistence. Every question, answer, coaching note, and table is hardcoded. Content cannot be updated without direct HTML editing.

→ Replace with: React frontend consuming a FastAPI content management API. All Q&A content stored in PostgreSQL. Educators author/edit via admin CMS, students consume dynamically rendered content.

Violation 02 Google Fonts External Dependency

The file loads Playfair Display, DM Sans, and DM Mono from Google Fonts CDN via a <link> tag. In a production app, this creates a GDPR/POPIA privacy issue (IP passed to Google) and a single point of failure.

→ Replace with: Self-hosted font files in the React app's /public/fonts/ directory. Use @font-face declarations in global CSS.

Violation 03 No User Identity or Access Control

Any person with the file URL can access all content — full model answers, writing coach tips, everything. There is no concept of a student account, a paid subscription gate, or an educator-only authoring view. Access is completely undifferentiated.

→ Replace with: JWT-based authentication via FastAPI. Role-based access control (RBAC) with Student, Educator, and Admin roles. Stripe or PayFast integration for premium content gating.

Violation 04 No Progress Tracking or Engagement State

The sticky nav shows which sections exist but provides no indication of which sections a student has read, highlighted, or bookmarked. There is no mechanism to track study completion, note personal annotations, or return to a specific reading position across sessions.

→ Replace with: PostgreSQL user_progress table tracking per-question read status, bookmarks, and notes. React state synced to backend on interaction.

Violation 05 Single-Module, Single-File — Not Scalable

The guide covers exactly one exam for one module (CAS3701). Scaling to multiple modules, multiple cohorts, multiple exam cycles, or multiple educators requires duplicating the entire HTML file and manually editing hardcoded content — a non-scalable, error-prone workflow.

→ Replace with: A module/exam/question hierarchical data model in PostgreSQL. Dynamic routing in React (/module/:id/exam/:id/question/:id). Educator CMS for multi-module management.

Violation 06 No Self-Assessment or Feedback Loop

The file presents model answers but provides no mechanism for students to test their understanding, submit practice responses, or receive feedback. The "Writing Coach Says" coaching layer is static text — it cannot respond to what a student actually writes.

→ Replace with: A self-assessment submission flow backed by FastAPI. Optional AI-powered feedback layer using the Anthropic API to evaluate student draft answers against rubric criteria.

Gap Analysis · Section 04

What It Needs

The following functional requirements are entirely absent from the current file and must be built from scratch to produce a real, production-ready EdTech platform:

  • 01
    Authentication & Authorization. JWT-based auth with bcrypt password hashing. Role definitions: Student, Educator, Admin. Social login optional (Google OAuth). Session refresh token rotation. Protected API routes per role.
  • 02
    PostgreSQL Data Model. Tables: users, modules, exams, questions, answers, coaching_notes, user_progress, user_notes, subscriptions, cohorts, cohort_members. Full relational schema with foreign key constraints and soft deletes.
  • 03
    FastAPI Backend — Content API. Routers: /auth, /modules, /exams, /questions, /progress, /notes, /admin. Pydantic schemas for all request/response shapes. SQLAlchemy ORM with Alembic migrations.
  • 04
    React Frontend Architecture. Pages: Login, Register, Dashboard, ModuleList, ExamView (question-by-question), NoteEditor, AdminCMS, AnalyticsDashboard. React Query for server state. Zustand or Context for auth state.
  • 05
    Educator Content Management System. Rich text editor (TipTap or Quill) for authoring model answers and coaching notes. Question builder with mark allocation fields. Draft/published states. Preview mode matching student view.
  • 06
    Student Self-Assessment Module. Per-question answer submission form. Optional timed mode. Draft save with autosave. Optional AI feedback integration (Anthropic API) for rubric-aligned answer evaluation.
  • 07
    Subscription / Access Control. Free tier (access to question text only) vs. Premium tier (model answers + coaching notes). Stripe integration for ZAR and USD billing. PayFast integration for South African market. Per-module or per-exam licensing options.
  • 08
    Progress Tracking. Per-student read status per question. Bookmark system. Personal annotations synced to backend. Study session timestamps. Exam readiness score calculated from self-assessment results.
  • 09
    Admin Analytics Dashboard. Active students per module, average time-on-page per question, self-assessment completion rates, subscription conversion rates. Exportable CSV report per cohort.
  • 10
    Security & Compliance. POPIA-compliant data handling (ironic given WellCore's POPIA discussion in the content). Input sanitisation, SQL injection prevention via ORM, rate limiting on auth endpoints (slowapi), CORS configuration, HTTPS enforcement.

Target Architecture · Section 05

Proposed Stack Structure

The target codebase follows a clean monorepo structure with separated /frontend and /backend directories, each independently deployable. Environment configuration is fully separated from code.

React Frontend

  • src/pages/ (route-level components)
  • src/components/ (shared UI)
  • src/features/ (domain modules)
  • src/services/ (API client layer)
  • src/hooks/ (data fetching)
  • src/store/ (auth + UI state)
  • src/styles/ (design tokens, globals)
  • React Router v6 for routing
  • React Query for server state
  • Axios for HTTP

FastAPI Backend

  • app/routers/ (API route handlers)
  • app/schemas/ (Pydantic models)
  • app/services/ (business logic)
  • app/repositories/ (data access)
  • app/models/ (SQLAlchemy ORM)
  • app/core/ (config, security, deps)
  • app/middleware/ (auth, logging)
  • Alembic for migrations
  • pytest for testing
  • slowapi for rate limiting

PostgreSQL Schema

  • users (id, role, email, hash)
  • modules (id, code, title, level)
  • exams (id, module_id, year, due)
  • questions (id, exam_id, num, marks)
  • answers (id, question_id, body, tips)
  • user_progress (user_id, question_id)
  • user_notes (user_id, question_id, text)
  • subscriptions (user_id, plan, stripe_id)
  • cohorts + cohort_members
  • audit_log (all write ops)

Environment Variables Required

Backend: DATABASE_URL, SECRET_KEY, ALGORITHM, ACCESS_TOKEN_EXPIRE_MINUTES, STRIPE_SECRET_KEY, PAYFAST_MERCHANT_ID, ANTHROPIC_API_KEY, CORS_ORIGINS
Frontend: VITE_API_BASE_URL, VITE_STRIPE_PUBLISHABLE_KEY
Infrastructure: Deployable to Railway, Render, or Fly.io with a managed PostgreSQL instance. Static frontend via Vercel or Netlify.

Step 03

FastAPI Auth Layer

Implement /auth/register, /auth/login, /auth/refresh, /auth/me endpoints. JWT token generation with RS256. Bcrypt password hashing. RBAC middleware. Rate limiting on auth routes with slowapi.

Step 04

Content API Routers

Build /modules, /exams, /questions, /answers CRUD routers with Pydantic schema validation. Implement access control — premium answers gated behind active subscription check. Add pagination and filtering to list endpoints.

Step 05

React Frontend — Auth + Routing

Build Login and Register pages with form validation (React Hook Form + Zod). Implement auth context with Zustand. Protected route wrapper component. React Router v6 with nested layouts (public vs. authenticated shell).

Step 06

ExamView Page — Core Study UX

Port the HTML design system into React components: QuestionCard, InsightBox, WritingTip, StyledTable, SWOTGrid, AnswerSection. Implement sticky section nav. Connect to content API via React Query. Match the original visual design fidelity.

Step 07

Progress Tracking & Annotations

Build /progress and /notes FastAPI endpoints. Implement per-question read tracking, bookmark toggle, and personal annotation editor on the frontend. Persist to PostgreSQL. Display progress indicator in the section nav.

Step 08

Educator CMS & Admin Dashboard

Build admin routes behind the Admin role guard. Implement TipTap rich-text editor for answer/coaching note authoring. Question builder form with mark field. Draft/publish workflow. Admin analytics page: student metrics, module engagement, subscription stats.

Step 09 (Optional — High Value)

AI Self-Assessment Layer

Integrate Anthropic API for student answer evaluation. Student submits draft answer → FastAPI sends to Claude with the model answer and rubric criteria → returns structured feedback JSON → frontend renders per-criterion feedback panel. Gate behind premium subscription.

Step 10

QA, Security Hardening & Deployment

End-to-end pytest coverage ≥ 80%. Playwright or Cypress E2E test for exam view and self-assessment flow. OWASP Top 10 review. HTTPS enforcement. PostgreSQL connection pooling (PgBouncer). Deploy to Railway (backend + DB) + Vercel (frontend). Configure monitoring with Sentry.

Go / No-Go Verdict · Section 09

Readiness Assessment

MEDIUM READINESS

The content is extraordinary. The design system is production-quality. The academic depth — IFRS 10/3, IFRS 15 bill-and-hold, S12E SBC tax treatment, POPIA compliance — is genuinely authoritative. What this is not, yet, is an app. It is a document. The entire backend, auth layer, data model, and interactive UX must be built from zero.

Primary Blockers

1. No backend whatsoever — zero FastAPI, zero API layer, zero database
2. No user identity — content is fully public, no monetisation possible
3. No content management — Ms. Lee must hand-edit HTML to update any answer
4. Single-module, single-exam — structurally unscalable without a database
5. No progress tracking, self-assessment, or student feedback loop

Effort Estimate

MVP (auth + content API + React ExamView + progress tracking): 180–240 hours
Full platform (CMS + admin + subscriptions + AI feedback): 380–520 hours
Team recommendation: 1 full-stack engineer + 1 UI engineer for MVP in 6–8 weeks. Add 1 backend specialist for AI integration sprint.
