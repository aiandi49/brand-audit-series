File Overview  ·  Section 01

What This File Is

Justice Path is a fully rendered, single-file HTML legal empowerment platform — a purpose-built case management tool pre-loaded with the real legal situations of two named individuals. The file presents as a password-gated dashboard with a left-nav sidebar, tabbed case views, an AI-powered legal chat interface, an automated legal document generator, a law library browser, and a new-case intake analyzer. The entire frontend is production-quality, lovingly styled, and functionally wired.

"This is the most socially impactful file in this audit series — a real legal operations platform for people who cannot afford an attorney. The mission is urgent, the design is complete, the backend does not exist."

Core product: A case-specific legal management dashboard that gives ordinary people — navigating real disputes against corporations, insurers, and government entities — the tools, contacts, documents, and AI counsel they need to fight back without a lawyer.

Implied user roles: Authenticated case clients (Erica, Luke — two real named users hardcoded). An implied admin role (absent) for onboarding new clients. General public users via the "Analyze New Case" tool.

User journey: Login → Case Dashboard → Action Plan → Generate Documents → AI Attorney Chat → File Complaints → Track Progress.

UI / Feature Inventory  ·  Section 02

What It Has

This file contains an unusually complete frontend for its class. The following features are visually and structurally present:

🔐 Login Screen

Full username/password UI with error messaging. Credentials are hardcoded in plaintext JavaScript. Two users: erica / erica1, luke / luke1. No session persistence.

📋 Case Overview Dashboard

Renders case summary, damages table, respondent list, legal theories, and jurisdiction info. All content is hardcoded per user in a JS CASES object — not fetched from any API.

🎯 Action Plan

Prioritized action items (URGENT / HIGH / MEDIUM) with descriptions, links to agency portals, and contact numbers. 9–10 actions per case. Entirely static.

📞 Contacts Directory

7 pre-populated contacts per case: attorneys general, regulatory boards, advocacy groups. Includes phone, email, and physical addresses. No click-to-call or email integration.

📄 AI Document Generator

6 document templates (Demand Letter, BBB Complaint, AG Complaint, Licensing Board, Small Claims, Media Statement). Each calls the Anthropic API live with the user's case context injected into the prompt. Copy-to-clipboard output. Functional — requires user-supplied API key.

🤖 AI Attorney Chat

Full conversational chat interface with multi-turn history. System prompt pre-loaded with complete case facts per user. Calls Anthropic API live. Working — requires user-supplied API key.

📚 Law Library

19 legal topics across 5 categories. Clicking any topic fires an Anthropic API call for a plain-English explanation. No static content — fully AI-driven. Requires API key.

➕ New Case Analyzer

Free-text intake form. User describes their situation; a structured AI analysis returns legal theories, agency contacts, prioritized actions, document recommendations, and statute of limitations warnings. Requires API key. The most generalizable feature in the app.

Technology & Dependency Audit  ·  Section 03

What Breaks the Requirements

Justice Path is remarkably clean of prohibited third-party platforms. There is no Firebase, no Databutton, no Riff, no vendor-locked backend. The violations are structural — not technological — but they are critical blockers for production deployment.

Plaintext Credentials in Client-Side JavaScript

The USERS object in the script block stores usernames and passwords in cleartext: erica: { password: 'erica1' }. Any user who opens DevTools sees every credential. This is a catastrophic security vulnerability — not a design oversight, a showstopper.

→ Replace with: FastAPI JWT auth endpoint (/auth/login), bcrypt-hashed passwords in PostgreSQL, HttpOnly cookie or Bearer token session management in React.
User-Supplied Anthropic API Key Passed in Browser

The app asks users to paste their personal Anthropic API key into a text field. Every AI feature (document gen, attorney chat, law library, case analysis) passes this key directly in browser fetch headers. This means the key is exposed in network logs, browser history, localStorage state, and any browser extension. It also means each user must provision their own API account — a non-starter for a real product.

→ Replace with: A FastAPI proxy endpoint (/api/ai/*) that holds the Anthropic API key server-side in an environment variable. React calls your own backend; your backend calls Anthropic. Users never touch the key.
All Case Data Hardcoded in JavaScript

Two complete legal cases — Erica Morales and Luke Ferguson — are defined as static JS objects inside the HTML file, including full names, incident details, dollar amounts, agency contacts, and legal theories. This is not a proof-of-concept shortcut; it is the only data layer in the app. Adding a third client requires editing source code and redeploying.

→ Replace with: PostgreSQL tables (cases, users, actions, contacts, documents). FastAPI CRUD endpoints. React fetches case data on load from /api/cases/{case_id}. Admin interface for onboarding new clients without touching code.
No Session Persistence — Login Resets on Refresh

The authentication is purely DOM-based: the login div hides and the app div shows. Refreshing the page resets everything. There is no session token, no localStorage strategy, no cookie — just CSS display toggling.

→ Replace with: JWT stored in HttpOnly cookie (secure, sameSite=strict). FastAPI dependency injection for protected routes. React auth context that re-validates on app mount.
No Case Progress Tracking or State Persistence

The Action Plan contains 9–10 prioritized action items per case, but there is zero ability to mark items complete, add notes, log complaint confirmation numbers, or track what has been filed. Every session starts from the same static state. For real legal cases with real deadlines, this is a critical missing layer.

→ Replace with: PostgreSQL action_items table with status enum (pending/in_progress/complete), notes field, completion_date, confirmation_number. FastAPI PATCH endpoint. React checkbox UI with optimistic updates.
Missing Functional Requirements  ·  Section 04

What It Needs to Become Real

⚖
Secure Authentication System — JWT-based auth with refresh tokens, bcrypt password hashing, role system (client / admin / attorney), rate-limited login endpoint, account recovery flow.
🗄
Full PostgreSQL Data Model — users, cases, actions, contacts, documents, generated_docs, law_library_topics, case_notes, audit_log. Alembic migrations for version control.
🔗
FastAPI AI Proxy Layer — Server-side Anthropic integration so no API key ever touches the browser. Streaming support for long document generation. Per-user usage tracking and rate limiting.
✅
Action Item Progress Tracking — Checkbox state persisted to DB, status transitions (pending → in progress → complete), deadline tracking with alert logic, confirmation number logging per complaint filed.
📁
Document Storage & History — Generated documents saved to DB (not just copied to clipboard). Version history. PDF export. Ability to re-open, edit, and regenerate. S3-compatible storage for attachments.
🛡
Admin Interface — Onboard new cases without editing source code. Manage users and their cases. View action progress across all clients. Dashboard for case statuses, outstanding deadlines, and generated document history.
📬
Notification System — Email alerts for upcoming deadlines (statute of limitations, notice of claim windows). Confirmation of filed complaints. Weekly action item summaries. FastAPI background tasks + SMTP or SendGrid.
🔒
Security Hardening — Input validation (Pydantic schemas), SQL injection prevention (SQLAlchemy ORM), CSRF protection, HTTPS enforcement, API rate limiting (slowapi), audit logging of all AI calls and document generations.
📱
Mobile Responsiveness — The layout collapses at small breakpoints but the sidebar and chat interface need mobile-specific reworking. Most users navigating a legal crisis will be on a phone.
⚠
Legal Disclaimer & Compliance Layer — Conspicuous "not legal advice" disclaimers per feature, user acknowledgment on first login, state-specific disclaimer logic for regulated legal information services.
Target Architecture  ·  Section 05

Proposed Final Structure

React Frontend — /frontend
  • src/pages/ — Login, Dashboard, Actions, Contacts, Documents, Attorney, Library, NewCase, Admin
  • src/components/ — CaseHeader, ActionItem, ContactCard, DocTile, ChatBubble, LibraryTopic, PriorityBadge, Spinner
  • src/services/ — api.js (Axios instance), auth.js, cases.js, ai.js, documents.js
  • src/context/ — AuthContext, CaseContext
  • src/hooks/ — useCase, useChat, useDocGen, useActions
  • React Router v6 with protected routes. Zustand or Context for global state. React Query for data fetching and caching.
FastAPI Backend — /backend
  • routers/ — auth.py, cases.py, actions.py, contacts.py, documents.py, ai.py, admin.py, library.py
  • schemas/ — Pydantic models for all request/response shapes
  • services/ — anthropic_service.py (proxy), document_service.py, email_service.py, auth_service.py
  • repositories/ — SQLAlchemy data access layer for each entity
  • models/ — User, Case, ActionItem, Contact, GeneratedDocument, LawTopic, CaseNote
  • FastAPI Depends() for auth injection. Background tasks for email notifications. Streaming response for AI generation.
PostgreSQL Schema — Core Tables
  • users (id, email, hashed_password, role, created_at)
  • cases (id, user_id, name, subtitle, state, case_type, summary, created_at)
  • damages (id, case_id, item, amount, order)
  • respondents (id, case_id, name, role)
  • legal_theories (id, case_id, theory)
  • action_items (id, case_id, priority, title, desc, link, contact, status, deadline, completion_date, notes)
  • contacts (id, case_id, name, title, phone, email, note)
  • generated_documents (id, case_id, user_id, doc_type, content, created_at)
  • chat_history (id, case_id, user_id, role, content, timestamp)
  • law_topics (id, category, name, cached_explanation, last_fetched)
Environment Variables
  • ANTHROPIC_API_KEY, DATABASE_URL, SECRET_KEY, ALGORITHM, ACCESS_TOKEN_EXPIRE_MINUTES
  • SMTP_HOST, SMTP_USER, SMTP_PASS, FROM_EMAIL, FRONTEND_URL
  • AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, S3_BUCKET (for document attachments)
Implementation Roadmap  ·  Section 06

Step-by-Step Build Plan

1
Scaffold the Monorepo
Create /frontend (Vite + React) and /backend (FastAPI) directory structure. Set up Docker Compose with postgres, backend, frontend services. Initialize Alembic for migrations. Establish .env strategy across environments.
2
PostgreSQL Schema + Migrations
Define all SQLAlchemy models. Run initial Alembic migration. Seed the database with Erica and Luke's case data extracted from the hardcoded JS objects. Confirm data integrity before building any API layer.
3
Authentication System
Build FastAPI /auth/register, /auth/login, /auth/refresh, /auth/logout endpoints. Implement JWT with HttpOnly cookies. Build React AuthContext with protected route wrappers. Convert the existing login UI to real auth.
4
Case Data API + React Pages
Build FastAPI CRUD endpoints for cases, actions, contacts, respondents, damages. Convert all hardcoded CASES JS objects to API calls. Wire React pages (Overview, Actions, Contacts) to live data. Action item status update endpoints.
5
AI Proxy Layer
Build /api/ai/chat, /api/ai/generate-document, /api/ai/explain-topic, /api/ai/analyze-case endpoints in FastAPI. Server holds ANTHROPIC_API_KEY. Strip the client-side API key input from the React UI. Implement streaming responses for document generation. Save all generated docs to DB.
6
Document History + Export
Wire document generation to persist output in generated_documents table. Build document history UI in the Documents tab. Add PDF export via weasyprint or pdfkit. Optional: S3 upload for permanent storage.
7
Admin Interface
React admin panel (role-gated) for creating new users and cases, editing action items and contacts, viewing all generated documents, and monitoring deadlines across the client base. FastAPI admin router with admin-only dependency.
8
QA, Security Hardening, Repo Readiness
Comprehensive Pydantic validation on all inputs. Rate limiting on AI endpoints. CORS configuration. E2E test suite (Playwright). README with quickstart, environment setup, and seeding instructions. GitHub Actions CI for lint + test. Deployment docs for Railway or Render.
Monetization Potential  ·  Section 07

The Business Case

$60k – $480k

A fully functional Justice Path competes in the legal tech / access-to-justice SaaS space — a sector with documented willingness to pay from nonprofits, legal aid organizations, law school clinics, community organizations, and individual litigants. $19–$49/month for individuals with active legal disputes. Free tier: new case analysis only. Paid tier: full case dashboard, document generator, AI attorney, deadline tracking. 200–500 paying users = $50k–$300k ARR. Churn risk: cases resolve (a good problem).

The Verdict

Strategic Note
The AI Attorney and Document Generator are genuinely differentiated. The New Case Analyzer alone has standalone SaaS potential. The mission framing unlocks nonprofit and government funding paths unavailable to most SaaS startups.
1
Eliminate the client-side API key immediately. Build a FastAPI /api/ai/* proxy that holds ANTHROPIC_API_KEY server-side. This single change removes the most dangerous vulnerability and unlocks the ability to control costs, rate-limit users, and track usage — which is the foundation of any billing model.
2
Extract all hardcoded case data to a PostgreSQL seed file. Define the full SQLAlchemy schema from the CASES JS object structure. Run Alembic migration. Seed with Erica and Luke's data. Now the app has a real data layer and onboarding a third client becomes an admin operation, not a code change.
3
Deploy the New Case Analyzer as a standalone public tool first. This feature requires no auth and no case data — just the AI proxy. A public-facing "Analyze Your Legal Situation" tool generates leads, demonstrates the product, and can go live in days. It is the fastest path to a real user in front of the product.
