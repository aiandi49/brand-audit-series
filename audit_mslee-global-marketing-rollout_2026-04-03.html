Section 03  ·  Technology & Dependency Audit

Violations & Constraints Check

This file is notably disciplined from a technical dependency perspective. The audit below covers every element against the hard requirements.

✓ CLEAN BASELINE: No Firebase. No Databutton. No Riff. No proprietary vendor services. No embedded third-party widgets. The file is a self-contained static document with zero backend coupling.

⚠ Item 01 — Dependency

Google Fonts via CDN (3 typefaces)

What it is: Playfair Display, DM Sans, and Space Mono loaded via fonts.googleapis.com.

Why noted: Not a violation — but introduces a network dependency that can cause FOUT (Flash of Unstyled Text) and fails in offline environments.

Self-host all three fonts in /frontend/public/fonts/. Use @font-face in global CSS. Vite bundles this automatically.

⚠ Item 02 — Hardcoded Content

All 7 Service Descriptions, 5 Personas, 6 Phases Are Inline HTML

What it is: Every service card, persona, timeline phase, channel tactic, KPI value, and budget row is hardcoded directly in the HTML with no data layer.

Why noted: Makes content updates require editing raw HTML. Not scalable for a live consulting business that adds services, updates pricing, or iterates on strategy.

All content becomes database records. FastAPI serves it via REST endpoints. React renders dynamically. Admin dashboard lets Ms. Lee update content without a developer.

⚠ Item 03 — Static Budget Table

$5,700 Budget Hardcoded With No Calculation Logic

What it is: The budget table is static HTML. Bar widths are hardcoded percentages (style="width:60%"). Totals are manually entered strings.

Why noted: A real budget planning tool needs dynamic calculation — change a monthly allocation and the 6-month total + percentage bars update automatically.

Build a React budget calculator component driven by a budget_allocations table in PostgreSQL. FastAPI exposes CRUD. Admin sets allocations, totals compute in real time.

⚠ Item 04 — Metric Counters Are UI Theater

KPI Values Animate but Are Not Connected to Real Data

What it is: The counter animation JS reads the existing HTML text content and animates it. "$80K" is a hardcoded string — not a live figure from a CRM or analytics source.

Why noted: In a live product, KPI cards should reflect actual performance: real email subscriber counts (from Mailchimp/Resend API), real course enrolments (from payment records), real follower counts (from social API). Currently the animation just "counts up to a lie."

FastAPI analytics endpoint aggregates real data from all integrated sources. React Query fetches and updates metric cards on a polling interval. Counter animation hooks into live values.

⚠ Item 05 — Voice AI & Quote Tool Are Implied, Not Present

Two Major Product Features Referenced but Not Implemented

What it is: The marketing plan heavily promotes a "Voice AI assistant" and a "Quote Tool" as primary lead magnets and differentiators. Neither exists in the HTML — they are referenced only in channel descriptions.

Why noted: These are the two highest-value conversion features in the entire plan. Without them, the marketing strategy loses its core "AI-powered differentiation" thesis entirely.

Voice AI: Integrate ElevenLabs or Vapi.ai API via FastAPI. Quote Tool: React multi-step form → FastAPI pricing logic → PostgreSQL quote records → automated email follow-up.

⚠ Item 06 — Vanilla JS Must Be Ported to React

scroll listener, IntersectionObserver, counter animation

What it is: Three distinct vanilla JS blocks handling UI behaviour.

Why noted: Not a violation — but cannot be left as inline scripts in a React app.

Scroll progress: CSS-only with animation-timeline: scroll() or a custom React hook. IntersectionObserver: react-intersection-observer library. Counter animation: react-countup or custom hook with useEffect.

✓ Net assessment: This file has zero prohibited platforms and zero technical debt in the traditional sense. Every "violation" is a consequence of the gap between a strategy document and a functional application — not of bad technical choices. The baseline is clean.

  • WhatsApp community of 500–1,000 members is referenced but not technically manageable from within the app

Security Requirements

Section 06  ·  Implementation Roadmap

Step-by-Step Build Plan

Step 01 — Week 1

Monorepo Scaffolding & CI/CD

Create /frontend (Vite + React 18 + React Query + React Router) and /backend (FastAPI + SQLAlchemy 2.0 + Alembic + Pydantic v2). Docker Compose for local dev. GitHub repo with main + dev branches. GitHub Actions CI: lint + test on every PR.

Step 02 — Week 1–2

Port HTML to React Component Library

Convert all 8 sections into React components. Replicate the design system (CSS variables, fonts, animations) using CSS Modules or Tailwind. Port all 3 JS behaviours to React hooks. Ensure pixel-perfect visual parity with the original HTML before adding any data layer.

Step 03 — Week 2

Define Data Contracts & Seed the Database

Write Pydantic v2 schemas for all entities. Create SQLAlchemy models. Run initial Alembic migration. Write a Python seed script that extracts all 7 service descriptions, 5 personas, 6 timeline phases, and 7 budget rows from the HTML into the database. After this step, the entire site renders from real data.

Step 04 — Week 2–3

Auth System

JWT auth with python-jose. Register, login, refresh, logout. Role-based access control middleware. Google OAuth2 via authlib. Frontend: AuthContext, protected routes, token storage in httpOnly cookies. Test: signup → login → access protected route → refresh → logout.

Step 05 — Week 3–4

Build the Quote Tool (Priority #1)

Multi-step React form: service pillar selection → qualifying questions → budget range input → contact info → submission. FastAPI pricing logic: maps selected services to estimate ranges stored in PostgreSQL. On submit: create quote record → send automated follow-up email via Resend → notify Ms. Lee via email/Slack. Admin view: see all submitted quotes with status management.

Step 06 — Week 4–5

Course Platform

Course listing page (from DB). Course detail with module breakdown. Stripe checkout for purchase. Enrollment record created on payment webhook. Course player: React video component using Mux Player. Progress tracking: POST /enroll/{course_id}/progress after each module. Certificate generation on 100% completion (PDF via reportlab).

Step 07 — Week 5–6

Voice AI Integration

React voice widget: microphone button → establishes a Vapi.ai session via FastAPI. FastAPI POST /voice/session: creates session, returns session token, stores interaction log. Voice agent scripts: FAQ routing, service inquiry qualification, discovery call booking via Calendly API. Post-session: FastAPI triggers email follow-up based on conversation outcome tags.

Step 08 — Week 6–7

Admin Dashboard & CMS

Role-gated /admin React pages. Service card CRUD (add/edit/reorder/toggle active). Blog post editor (rich text via TipTap). Quote inbox with status pipeline (new → reviewed → proposal sent → closed). Live KPI dashboard pulling from GET /admin/kpis — real subscriber count, enrollment count, quote volume, revenue. Budget planner with live recalculation.

Step 09 — Week 7–8

Email Sequences & WhatsApp Integration

5-email welcome sequence triggered on signup (Resend templates). Quote follow-up sequence (3 emails over 7 days if no response). Course onboarding email on enrollment. WhatsApp Business API: automated community welcome message on sign-up, weekly tip drops from admin dashboard. All sequences stored and editable in admin panel.

Step 10 — Week 8–9

Events, Webinars & Referral System

Events page: upcoming webinars from DB. Registration flow with email confirmation + Zoom link delivery. Zoom API integration for automated meeting creation on event publish. Referral system: unique referral codes per user → referrals table → reward logic (discount on next course, credit toward retainer). Admin: manage events, view registrations, export attendee list.

Step 11 — Week 9–10

QA, Performance & Compliance

Full end-to-end test suite: pytest (backend) + Playwright (frontend). Lighthouse score ≥ 90 on all pages. POPIA consent checkbox + audit log on all forms. Stripe test mode → production flip. Multi-region test: South Africa and Thailand routing. Mobile responsive audit (critical for WhatsApp-first markets). Load test core endpoints.

Step 12 — Week 10

Repo Readiness & Launch

README with full local setup, environment variable documentation, seeding instructions, and deployment guide. GitHub Actions CI/CD: auto-deploy to Vercel (frontend) and Fly.io (backend) on merge to main. Tag v1.0.0. Launch checklist: custom domain, SSL, Stripe live keys, email domain verification (SPF/DKIM/DMARC), analytics live.

Section 09  ·  Go / No-Go Verdict

MEDIUM

Readiness Rating: Medium

The business model is fully articulated, the design is production-quality, and the revenue thesis is well-reasoned. The engineering gap is significant — but every gap is well-defined, standard, and buildable without exotic technology.

Primary Blockers

  • No backend whatsoever — the entire API layer, auth system, and database are absent
  • Quote Tool and Voice AI — the two flagship product features — do not exist in any form
  • Course platform requires video hosting infrastructure (Mux) and payment integration (Stripe) both of which require account setup and API keys before a single line of product code can ship
  • Multi-region complexity: POPIA compliance (South Africa), multi-currency, and multilingual Voice AI add non-trivial scope to an already large build
  • WhatsApp Business API access requires Meta approval — apply immediately as this takes 1–4 weeks

Strengths to Leverage

  • The business strategy is the most detailed and considered of any file in this audit — the marketing plan itself is a highly sophisticated brief that eliminates ambiguity for engineers
  • Existing HTML design system is production-quality and translates cleanly into React components
  • Seven service lines means seven independent revenue streams — any one of them can be v1 while the others are built in parallel
  • Founder's authentic personal brand gives the marketing machine genuine fuel that ad spend cannot buy

Effort Estimate

Total estimated effort: 180–280 hours for a single senior full-stack developer, or a team of 2–3 across 6–10 sprints. This is a large build — the most complex file in this audit series — driven by the multi-service, multi-region, multi-channel product scope. Recommended approach: build a v1 "thin slice" (auth + services + quote tool + one course) and launch within 4 weeks, then build the remaining features behind a real paying user base.

  • Action 2 — Build the Quote Tool as the first shipped feature. It requires no auth, no video infrastructure, and no third-party approvals.   • Action 3 — Scaffold the monorepo and seed the database from the HTML today. The 7 services, 5 personas, 6 timeline phases, and budget rows in this HTML file are the product's first content. Extract them into a Python seed script, get them into PostgreSQL, and make the homepage render from real data within the first 3 days.
